# Find Security Bugs 常用规则语法

## 这份文档回答什么

这份文档不讲“有哪些漏洞”，而是专门讲:

1. FindSecBugs 规则文件常见长什么样
2. 每种配置语法各字段代表什么
3. 你要自定义 source、sink、safe encoder 时，最常见的写法是什么

如果你是新手，先记住一句:

`FindSecBugs 的常用规则语法，主要不是 YARA/正则那种语言，而是几类“方法签名 + taint 状态/参数索引/标签”的文本配置格式。`

## 先分清三大类常用语法

### 第一类: taint 配置语法

文件位置常见于:

`taint-config/*.txt`

作用:

描述某个方法返回值或参数的 taint 行为。

### 第二类: sink 配置语法

文件位置常见于:

`injection-sinks/*.txt`

作用:

描述某个方法的哪些参数是危险汇点。

### 第三类: safe encoder 配置语法

文件位置常见于:

`safe-encoders/*.txt`

作用:

描述某个方法会给数据增加安全标签，或者移除某些风险标签。

## 所有语法的共同基础: JVM 方法签名

不管是哪类文件，最常见的左半部分都长这样:

`包/类.方法(参数签名)返回值签名`

例如:

`javax/servlet/ServletRequest.getParameter(Ljava/lang/String;)Ljava/lang/String;`

如果你第一次看到这类写法会头疼，可以先这样拆:

1. `javax/servlet/ServletRequest` 是类
2. `getParameter` 是方法
3. `(Ljava/lang/String;)` 表示参数列表里有 1 个 `String`
4. `Ljava/lang/String;` 表示返回值也是 `String`

### 常见类型缩写

1. `Ljava/lang/String;` 表示 `java.lang.String`
2. `I` 表示 `int`
3. `Z` 表示 `boolean`
4. `V` 表示 `void`
5. `[I` 表示 `int[]`
6. `[Ljava/lang/String;` 表示 `String[]`

这类格式来自 JVM 字节码签名，不是 FindSecBugs 自创语法。

## 语法一: taint 配置

在 `taint-config/java-ee.txt` 里，可以看到非常典型的例子。

### 最简单写法

```text
javax/servlet/ServletRequest.getParameter(Ljava/lang/String;)Ljava/lang/String;:TAINTED
```

含义:

`ServletRequest.getParameter()` 的返回值视为 tainted。

### 另一个简单写法

```text
javax/servlet/ServletRequest.getProtocol()Ljava/lang/String;:SAFE
```

含义:

`getProtocol()` 返回值可视为安全。

### 参数传播写法

```text
javax/servlet/ServletRequest.getRealPath(Ljava/lang/String;)Ljava/lang/String;:0
```

含义:

返回值 taint 状态来自第 `0` 个参数。

用大白话说:

“你传进来的路径如果脏，返回值也跟着脏；如果你传进来的路径安全，返回值也继承这个状态。”

## taint 状态里最常见的关键字

源码里可见的核心状态包括:

1. `TAINTED`
2. `UNKNOWN`
3. `SAFE`
4. `NULL`
5. `INVALID`

对一般用户最常用的是前三个。

### `TAINTED`

表示:

不可信、应谨慎传播。

### `SAFE`

表示:

可认为安全，通常不会触发对应注入型告警。

### `UNKNOWN`

表示:

工具暂时不能肯定安全，也不能肯定危险，会采取更保守的传播策略。

## taint 配置里的标签语法

在 `TaintMethodConfig.java` 和 `safe-encoders/*.txt` 里能看到标签语法。

典型例子:

```text
java/net/URLEncoder.encode(Ljava/lang/String;)Ljava/lang/String;:0|+URL_ENCODED,+XSS_SAFE
```

拆解:

1. `:0` 表示返回值继承第 0 个参数的 taint 状态
2. `|+URL_ENCODED,+XSS_SAFE` 表示额外给结果打上两个正向标签

也就是说:

“这个值虽然仍与输入有关，但现在已经过 URL 编码，并且在 XSS 语境下更安全。”

### 去标签写法

```text
java/net/URLDecoder.decode(Ljava/lang/String;)Ljava/lang/String;:0|-URL_ENCODED,-XSS_SAFE
```

含义:

解码以后，原来那些“已编码”“XSS_SAFE”标签就不该继续保留了。

## `|` 和 `+`、`-` 的含义

### `|`

表示进入标签修改部分。

### `+TAG`

给结果增加标签。

### `-TAG`

从结果中移除标签。

## `#` 语法是干什么的

在 `TaintMethodConfig.java` 的正则里还能看到 `#` 段。

它表示:

方法会修改哪些栈位置对应的对象或参数，也就是“这个调用不仅返回值会变，输入对象本身也可能被改变”。

这是更偏引擎内部传播精度的高级语法。

对一般使用者来说:

1. 看懂它存在即可
2. 真要自定义时，再参照源码和现有配置做最小修改

## 语法二: sink 配置

在 `injection-sinks/sql-jdbc.txt` 里，最典型格式是:

```text
java/sql/Statement.executeQuery(Ljava/lang/String;)Ljava/sql/ResultSet;:0
```

含义:

`executeQuery` 的第 `0` 个参数是危险汇点参数。

如果 tainted 数据流到这个位置，就可能形成 `SQL_INJECTION_JDBC`。

### 多参数写法

```text
java/sql/Statement.execute(Ljava/lang/String;I)Z:1
```

这里表示:

这个方法被认为第 `1` 个参数位是需要关注的危险位置。

### 多个危险参数

语法支持:

```text
某方法签名:0,2
```

含义:

第 0 和第 2 个参数都可能是 sink。

## sink 文件的语法规则很朴素

从 `SinksLoader.java` 可知:

1. 一行只允许一个 `:`
2. 冒号左边是完整方法签名
3. 冒号右边是一个或多个参数索引
4. 参数索引之间用逗号分隔
5. 空行跳过
6. 以 `-` 开头的行跳过

这意味着 sink 文件本质上很接近:

“方法签名 -> 哪些参数危险”

## 语法三: safe encoder 配置

在 `safe-encoders/other.txt` 里，常见格式是:

```text
org/springframework/web/util/HtmlUtils.htmlEscape(Ljava/lang/String;)Ljava/lang/String;:0|+XSS_SAFE
```

含义:

返回值继承第 0 个参数的数据来源关系，但额外增加 `XSS_SAFE` 标签。

也就是:

“它仍来自用户输入，但因为被 HTML escape 过，所以在 HTML 文本上下文里更安全。”

另一个很重要的例子:

```text
java/sql/Statement.enquoteLiteral(Ljava/lang/String;)Ljava/lang/String;:0|+SQL_INJECTION_SAFE
```

含义:

这个返回值与输入相关，但在 SQL 注入语境下被认为更安全。

## 注释和禁用行的写法

这些 txt 规则文件里，经常会看到:

1. 空行
2. `-` 开头的行
3. `--` 开头的人类说明行

引擎处理时，凡是 `-` 开头的行都会跳过。

所以你可以把它简单理解成:

`-` 开头就是注释或临时禁用。

## 自定义规则时最常见的 4 个任务

### 1. 增加新的 source

例如公司自己封装了:

`com/acme/web/Input.getUserParam(Ljava/lang/String;)Ljava/lang/String;`

你可以写:

```text
com/acme/web/Input.getUserParam(Ljava/lang/String;)Ljava/lang/String;:TAINTED
```

### 2. 增加新的 sink

例如公司自己封装了:

`com/acme/db/Exec.raw(Ljava/lang/String;)V`

你可以写:

```text
com/acme/db/Exec.raw(Ljava/lang/String;)V:0
```

### 3. 增加新的 safe encoder

例如公司自己实现了统一 HTML 编码器:

```text
com/acme/security/Encode.html(Ljava/lang/String;)Ljava/lang/String;:0|+XSS_SAFE
```

### 4. 去掉不该保留的标签

如果某个解码函数会把安全编码还原，可以写成:

```text
com/acme/security/Decode.html(Ljava/lang/String;)Ljava/lang/String;:0|-XSS_SAFE
```

## 初学者最容易写错的地方

### 1. 类名分隔符写成点号

规则文件常见的是:

`java/sql/Statement`

不是:

`java.sql.Statement`

### 2. 方法签名少了返回值

JVM 方法签名必须完整，不能只写方法名和参数。

### 3. 参数索引理解错位

一定要按该规则文件和底层解析逻辑理解参数位，不要凭源码表面位置想当然。

### 4. 把 sink 语法和 taint 语法混写

记忆法:

1. taint 文件关注“返回值/状态/标签”
2. sink 文件关注“哪几个参数危险”

### 5. 不做最小验证

每写一条自定义规则，都最好配一个最小样例 class 验证是否真的命中。

## 给非科班读者的类比

可以把这些规则文件想成医院分诊表:

1. source 规则说“哪些病人来自高风险区域”
2. sink 规则说“哪些科室绝不能让高风险病人直接进入”
3. safe encoder 规则说“哪些消毒流程可以降低风险”

这就是为什么一条规则并不是孤零零的一行文本，而是一整套关联定义。

## 最小记忆卡片

只记最实用的 5 条:

1. `方法签名:TAINTED` 表示返回值不可信
2. `方法签名:SAFE` 表示返回值安全
3. `方法签名:0` 表示返回值继承第 0 个参数 taint
4. `方法签名:0|+XSS_SAFE` 表示继承 taint 并增加安全标签
5. `方法签名:0,2` 在 sink 文件里表示第 0 和第 2 个参数是危险点
