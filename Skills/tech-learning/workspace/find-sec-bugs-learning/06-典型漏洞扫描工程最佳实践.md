# 使用 Find Security Bugs 做典型漏洞扫描工程的最佳实践

## 这份文档回答什么

这份文档不是讲“工具原理”，而是讲“怎么把它用对”:

1. 扫描一个真实 Java 工程时，推荐的流程是什么
2. 怎样降低误报、减少漏报
3. 怎样把它接进 CI/CD
4. 怎样做基线、分级、回归
5. 怎样在团队里落地，而不是只跑一次命令就结束

如果你只记一句话:

`FindSecBugs 最好的用法，不是“偶尔跑一下”，而是把它当成 Java 安全静态分析流水线中的常驻体检项。`

## 最重要的前提: 它扫描的是编译产物

所以一次合格的工程化扫描，通常从“先把项目编译出来”开始。

推荐输入优先级:

1. `target/classes`
2. `build/classes`
3. 可部署 `jar/war`
4. 多模块统一产物目录

不推荐把“源码目录”当主入口。

## 最小可落地流程

### 第 1 步: 固定扫描版本

要固定:

1. SpotBugs 版本
2. FindSecBugs 插件版本
3. Java 运行时版本

原因:

如果这三项漂移，你很难判断告警变化到底来自:

1. 代码变化
2. 规则变化
3. 运行时变化

### 第 2 步: 在 CI 里先编译

例如:

1. Maven: `mvn -DskipTests package`
2. Gradle: `./gradlew classes` 或 `./gradlew build`

### 第 3 步: 用固定命令扫描

例如:

```bash
findsecbugs -textui -maxHeap 4096 target/classes
```

### 第 4 步: 保存结构化结果

最好不要只看终端滚屏，而要把结果保存成可归档、可比较的格式。

原因:

1. 便于做历史基线
2. 便于在 PR 和版本之间比对
3. 便于交给内部平台统一聚合

### 第 5 步: 对结果分级

不要把所有结果一刀切。

至少分成:

1. 高危立即处理
2. 中危排期修复
3. 低危观察或人工复核
4. 已确认误报纳入基线

## 真实项目里最推荐的 10 条实践

### 1. 先扫主产物，不要一上来全仓乱扫

对大型仓库，先确定“真正对外提供功能的模块”。

这样做的好处:

1. 先收获高价值结果
2. 减少噪音
3. 更快建立团队信任

### 2. 保证依赖尽量齐全

静态分析不是只看你自己的类。父类、接口、第三方框架类型信息经常影响分析精度。

缺依赖时常见后果:

1. 一些调用关系识别不准
2. 某些框架入口无法识别
3. 漏报和噪声都可能增加

### 3. 不要直接用“零容忍”阻断全量历史告警

首次落地时，老项目往往一口气出来很多结果。

正确顺序通常是:

1. 先跑全量
2. 人工审一轮
3. 建立基线
4. 后续只阻断新增高危问题

### 4. 给公司自定义框架补 source/sink/safe encoder

这通常是提升效果最划算的一步。

原因:

你的业务代码经常不会直接调用原生框架 API，而是经过内部封装。

如果不补配置，工具可能:

1. 看不见真实入口
2. 看不见真实危险 API
3. 认不出你们自己的净化函数

### 5. 每加一条自定义规则，都配一个最小复现样例

这是最容易被忽视、却最值钱的一条。

建议每个自定义规则至少准备:

1. 一个应该命中的 bad case
2. 一个不应命中的 good case

这样你升级版本时就能回归验证。

### 6. 把“高价值漏洞族”优先纳入治理

如果团队精力有限，优先关注:

1. `SQL_INJECTION_*`
2. `COMMAND_INJECTION`
3. `PATH_TRAVERSAL_*`
4. `XSS_*`
5. `XXE_*`
6. `OBJECT_DESERIALIZATION`
7. `WEAK_TRUST_MANAGER`
8. `HARD_CODE_PASSWORD`

这些通常更容易转化为真实安全收益。

### 7. 用代码审计思维看结果，不要把工具当裁判

静态扫描结果更像“线索”，不是最终判决书。

建议审计人员看每条结果时至少回答:

1. 数据真的可控吗
2. 危险 API 真能执行到吗
3. 中间是否已有足够净化
4. 漏洞利用条件是否成立

### 8. 大项目要提前给足内存

经验上，Java 安全扫描很容易吃内存。

保守建议:

1. 小项目: 默认或 `2048`
2. 中项目: `4096`
3. 大型单体或多模块: 更高

### 9. 把告警治理流程写进团队制度

如果没有制度，工具很快会变成“一次性演示”。

至少应定义:

1. 谁看结果
2. 谁确认误报
3. 谁决定阻断门槛
4. 谁维护自定义规则
5. 升级版本时谁做回归

### 10. 升级前做对比回归

升级 SpotBugs 或 FindSecBugs 时，不要直接替换生产扫描器。

推荐做法:

1. 对同一批项目跑新旧两版
2. 看新增告警、消失告警、优先级变化
3. 评估是否是规则增强还是噪音增长

## 典型落地场景建议

### 场景 1: 小团队第一次引入

推荐策略:

1. 只扫 1 到 3 个核心服务
2. 只盯高危家族
3. 建立第一版误报基线
4. 每周固定复盘一次

目标:

先证明价值，而不是一开始覆盖全公司。

### 场景 2: 安全团队批量审计多个 Java 仓库

推荐策略:

1. 统一 Java 和扫描器版本
2. 统一输出格式和目录结构
3. 统一高危规则分级标准
4. 对常见内部框架补一版共享自定义规则

目标:

减少仓库间口径不一致。

### 场景 3: 已有 CI，希望纳入 PR 阶段

推荐策略:

1. 全量扫描继续在 nightly 或主分支跑
2. PR 阶段只看新增高危
3. 首期不要让低危和历史问题阻断合并

目标:

在不严重拖慢研发的前提下提高门槛。

## 如何降低误报

最有效的办法通常不是“关掉规则”，而是提高上下文质量。

优先顺序建议如下:

1. 保证依赖完整
2. 补充自定义 safe encoder
3. 补充真实 source/sink 定义
4. 对个别规则做有依据的过滤
5. 用人工复核建立可追溯基线

## 如何降低漏报

最有效的办法通常是补业务知识。

例如:

1. 自定义控制器入参来源
2. 自定义 SQL 执行器
3. 内部模板渲染器
4. 自定义命令执行封装

对 FindSecBugs 来说，很多漏报不是“引擎完全不行”，而是“它不知道你公司内部的 API 语义”。

## 推荐的结果治理模型

一个简单、实用的治理模型可以是:

### `新增高危`

默认阻断或要求安全确认。

### `新增中危`

进入开发排期或安全评审。

### `历史存量`

建立基线，逐步消化，不立即阻断。

### `误报`

必须记录原因，不要口头忽略。

## 对非科班团队的落地建议

如果团队里很多人不是安全背景，也不是编译原理背景，最有效的推进方式通常不是讲术语，而是讲“固定动作”。

建议把流程固化成 5 件事:

1. 代码先编译
2. 扫描器固定版本
3. 每次结果存档
4. 新增高危优先看
5. 公司内部框架持续补规则

这样即使成员不懂 taint 分析细节，也能把工具稳定用起来。

## 最后一个常被忽略的实践: 把学习材料和扫描器放一起维护

很多团队失败，不是因为工具不好，而是因为只有一个人懂。

所以推荐把下面这些一起版本化维护:

1. 扫描脚本
2. 自定义规则
3. 误报基线
4. 结果分级标准
5. 学习文档

这样人员变动后，工程能力不会断层。

## 一句话总结

`FindSecBugs 最强的价值，不是“神奇地自动发现一切漏洞”，而是把 Java 安全审计中最重复、最适合规模化的那部分工作工程化。`
