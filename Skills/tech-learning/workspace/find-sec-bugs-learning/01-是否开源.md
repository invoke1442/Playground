# Find Security Bugs 是否开源

## 这份文档回答什么

这份文档专门回答一个看似简单、但实际很重要的问题:

1. `Find Security Bugs` 到底是不是开源软件。
2. 它开放到了什么程度。
3. 你可以怎样合法地使用、修改、再分发它。
4. 它和“免费但不开源”“源码可看但不让改”的工具有什么区别。

如果你不是科班出身，可以先记住一句话:

`Find Security Bugs` 是开源的，而且是公开维护的安全静态分析项目，不只是“可以免费下载”，而是真正提供源码、许可证、构建脚本、规则定义、测试样例和发布流程。

## 结论先说

`Find Security Bugs` 是开源项目。

判断依据不是“网上很多人都这么说”，而是下面这些可以直接核验的事实:

1. 它有公开仓库: `https://github.com/find-sec-bugs/find-sec-bugs`
2. 仓库里包含许可证文件: `LICENSE`、`LGPL-3.0.txt`
3. 仓库 README 明确写了许可证: `This software is release under LGPL.`
4. 源码、规则、样例、构建脚本都在仓库中公开可见
5. 官方网站也明确写了项目是 open-source，并欢迎贡献

## 开源不等于“随便拿来闭源”

README 和许可证文件表明它采用的是 `LGPL`，也就是 `GNU Lesser General Public License`。

对非法律背景的人，可以把它理解成下面这套近似规则:

1. 你可以使用它。
2. 你可以研究它怎么实现。
3. 你可以修改它。
4. 你可以分发原版或修改版。
5. 但你分发它本身或基于它修改后的库时，通常需要保留相应的开源许可证义务。

最重要的现实理解是:

1. 它不是私有商业闭源扫描器。
2. 它不是“只能看不能改”的源码开放。
3. 它是允许社区参与、允许派生修改、允许提交贡献的真正开源项目。

## 为什么我能确定它不是“伪开源”

很多工具虽然把一部分代码放到 GitHub，但核心规则、发布流程、测试或许可证不完整，这种项目在工程上不一定算“可用的开源”。

`Find Security Bugs` 不是这种情况。原因有四类。

### 1. 有完整源代码

核心插件源码在:

`findsecbugs-plugin/src/main/java/`

这里不仅有几个示例类，而是完整检测器实现，例如:

1. `crypto/WeakMessageDigestDetector.java`
2. `injection/sql/SqlInjectionDetector.java`
3. `taintanalysis/TaintDataflowEngine.java`

这意味着你不是只能“调用”规则，而是能读到规则到底怎么判断。

### 2. 有完整规则元数据

在:

`findsecbugs-plugin/src/main/resources/metadata/findbugs.xml`

这个文件里，项目显式注册了:

1. 检测器 `Detector`
2. 漏洞模式 `BugPattern`
3. 引擎注册器 `EngineRegistrar`

我本地统计到:

1. `121` 个检测器注册项
2. `144` 个漏洞模式注册项

这类元数据公开，说明规则不是黑盒。

### 3. 有完整规则知识库文件

例如:

1. `taint-config/*.txt`
2. `injection-sinks/*.txt`
3. `safe-encoders/*.txt`

这些文件定义了:

1. 哪些 API 产生污染数据
2. 哪些 API 是危险汇点
3. 哪些 API 可以视为安全编码器

也就是说，连“数据流规则表”都是开着的。

### 4. 有样例工程和 CI

仓库里有多套样例项目:

1. `findsecbugs-samples-java`
2. `findsecbugs-samples-java11`
3. `findsecbugs-samples-jsp`
4. `findsecbugs-samples-kotlin`

还有 GitHub Actions 工作流。

这说明它不是“扔一个仓库出来就不管”，而是有持续维护痕迹。

## 对普通使用者意味着什么

如果你只是把它当扫描器来用，开源至少带来六个现实价值。

### 1. 结果更可解释

你看到一条 `SQL_INJECTION_JDBC` 告警时，不必靠猜。你可以回到源码和配置文件，看它为什么会报。

### 2. 可以审计误报和漏报

如果你觉得某条规则报多了，或者某个框架没覆盖，你可以直接检查:

1. 污染源配置有没有你的框架方法
2. sink 配置有没有你的危险 API
3. safe encoder 配置有没有你的净化函数

### 3. 可以做企业内部定制

比如公司内部封装了 `CustomSqlExecutor.execute(String sql)`，默认规则可能不认识。你就可以通过自定义 sink 配置把它纳入检测范围。

### 4. 可以做二次集成

由于规则、CLI、Maven 坐标都公开，你可以集成到:

1. CI
2. 代码审计平台
3. IDE
4. 批量扫描流水线

### 5. 学习成本更可控

很多安全工具只给“功能说明”，不给“原理说明”。而开源项目允许你边用边学，逐步建立自己的判断标准。

### 6. 可以自己修

发现 bug 时，你不一定要等厂商。只要能力够，可以自己修并提交 PR，或者先在内部维护补丁版。

## 它开源，但不代表“任何地方都一样新”

需要注意一个常见误区:

开源项目的“仓库源码版本”“Maven 发布版本”“你本机安装的二进制版本”不一定完全相同。

我本地看到:

1. 你指定的源码仓库 `pom.xml` 是 `1.15.0-SNAPSHOT`
2. 我之前安装并验证可用的本地二进制插件版本是 `1.14.0`
3. 本地搭配的 SpotBugs 版本是 `4.9.8`

这不影响它是否开源，但会影响你看到的规则数量、行为和文档对应关系。

所以实际学习时要区分三件事:

1. 仓库主线在开发什么
2. 官方最近发布了什么
3. 你本机实际跑的是哪一版

## 怎样用最朴素的方法判断一个安全工具是否开源

以后你看到别的 SAST 工具，也可以套这套检查表:

1. 有没有公开仓库
2. 有没有明确许可证
3. 有没有完整源码而不是只放接口
4. 有没有构建脚本
5. 有没有规则定义或插件机制
6. 有没有测试样例
7. 有没有发布说明和 issue/PR 活跃度

`Find Security Bugs` 在这套检查表上是合格的。

## 一个适合非科班用户的类比

你可以把它想成“开源的安检机”:

1. 机器本身公开了图纸
2. 检测规则公开了
3. 样品和测试流程也公开了
4. 你可以自己加一条“这类包裹也要重点查”的规则

这和“机场只卖一个黑箱子，告诉你它能查危险品，但不告诉你怎么查”的闭源方案，是本质不同的。
