# Find Security Bugs 的静态分析原理与工作流

## 这份文档回答什么

这份文档回答:

1. FindSecBugs 为什么能在不运行程序的情况下发现漏洞
2. 它到底分析的是源码、字节码，还是别的东西
3. taint 分析是怎么工作的
4. 一次扫描从输入到输出，内部大致经历哪些步骤

如果你完全没学过编译原理，也没关系。你只要先记住一句话:

`FindSecBugs 不是去“运行你的程序”，而是把编译后的 Java 类文件当作地图，沿着可能的数据流路径追踪“不可信数据会不会流到危险位置”。`

## 什么叫静态分析

静态分析的反义词是“动态分析”。

### 动态分析

程序真的跑起来，然后观察它做了什么。

像什么:

1. 跑接口测试
2. 跑模糊测试
3. 实时抓网络流量

### 静态分析

程序不跑，只分析程序结构本身。

像什么:

1. 看 class 文件
2. 看调用关系
3. 看某个输入有没有可能最终拼进 SQL

FindSecBugs 属于后者。

## 它主要分析什么材料

核心答案:

它主要分析 `Java 字节码`，也就是编译后的 `.class`、`.jar`、`.war` 等。

这也是为什么很多文档和实践都会让你传:

1. `target/classes`
2. `build/classes`
3. `jar`
4. `war`

而不是直接传 `src/main/java`。

## 为什么它偏爱字节码而不是源码

因为 Java 世界里，不同构建工具、不同语言方言、不同注解处理器，会让源码层差异很大。

而字节码更统一。

统一带来三件好事:

1. 可以跨 Maven、Gradle、Ant
2. 可以支持 Java、部分 Kotlin、Scala 等最终落到 JVM 的产物
3. 可以基于稳定的调用指令做数据流分析

## 它的分析核心不是正则，而是“规则 + 字节码 + 数据流”

很多初学者以为安全扫描器就是:

“看到某个关键字就报警。”

FindSecBugs 不是这么简单。

它至少有两种主要工作方式。

### 第一类: 直接模式匹配型检测

例如弱哈希、弱加密这类规则。

典型例子:

`WeakMessageDigestDetector.java`

它会检查:

1. 是否调用了 `MessageDigest.getInstance`
2. 传进去的常量是不是 `MD2`、`MD4`、`MD5`、`SHA1`

这类规则比较像“带语义的模式匹配”。

### 第二类: 污染传播型检测

例如:

1. SQL 注入
2. 命令注入
3. LDAP 注入
4. XSS
5. 路径遍历

这类规则更复杂。它们通常不是看到一个方法名就直接报，而是要追踪:

1. 数据从哪来
2. 中间经过了什么处理
3. 最后有没有流到危险 API

## taint 分析是什么

可以把 `taint` 直译成“污染标记”。

### 最朴素的理解

如果一段数据来自用户、网络、Header、Cookie、URL、表单，它就可能不可信。

工具会先给它贴一个标签:

`这东西可能脏。`

然后继续跟踪:

1. 它有没有被复制
2. 有没有拼接到别的字符串里
3. 有没有被某个安全编码函数处理
4. 最后有没有进 SQL、命令执行、文件路径、HTML 输出等危险位置

如果有，就报漏洞。

## 在源码里，它的 taint 引擎是怎样接上的

在 `metadata/findbugs.xml` 里，插件注册了:

`EngineRegistrar`

对应类:

`com.h3xstream.findsecbugs.taintanalysis.EngineRegistrar`

而真正执行方法级数据流分析的类是:

`TaintDataflowEngine.java`

这个类会:

1. 预加载多个 taint 配置文件
2. 预加载安全编码器配置
3. 根据系统属性补充自定义配置
4. 为每个方法构造控制流图和搜索结构
5. 运行 `TaintAnalysis`
6. 得到 taint dataflow 结果

## 一次扫描的大致工作流

下面这条流程，适合普通读者理解全局。

### 第 1 步: 你把 class/jar 交给 CLI

例如:

```bash
findsecbugs -textui target/classes
```

### 第 2 步: CLI 启动 SpotBugs

包装脚本会把:

1. SpotBugs jar
2. FindSecBugs 插件 jar
3. 过滤文件 `include.xml`

一起喂给 Java 进程。

### 第 3 步: SpotBugs 读取字节码

它会解析类、方法、常量池、调用指令等。

可以把这一步理解为:

“把编译后的程序拆成机器可分析的结构化表示。”

### 第 4 步: 插件注册的检测器开始参与

SpotBugs 加载 `findbugs.xml` 中声明的检测器。

不同检测器关注不同事情:

1. 加密类看加密 API
2. XML 类看解析器安全配置
3. 注入类看 source 到 sink 的路径

### 第 5 步: taint 引擎加载知识库

`TaintDataflowEngine` 会把这些文件装入内存:

1. `taint-config/*.txt`
2. `safe-encoders/*.txt`
3. 可选自定义 taint 配置

这些文件告诉它:

1. 什么是脏数据
2. 什么能保持脏
3. 什么可以洗干净

### 第 6 步: 构建方法级数据流图

引擎会拿到:

1. `CFG` 控制流图
2. `DepthFirstSearch`
3. `MethodGen`

然后在方法内推演:

1. 当前栈上有哪些值
2. 这些值是否 tainted
3. 标签是否变化

### 第 7 步: 命中危险汇点

注入检测器会查询 sink 表。

例如 `sql-jdbc.txt` 里定义:

1. `Statement.executeQuery(String)` 的第 0 个参数是危险点
2. `Connection.prepareStatement(String)` 的第 0 个参数是危险点

如果 tainted 数据流到了这些位置，就可能形成漏洞告警。

### 第 8 步: 应用安全标签修正优先级

如果中间经过了某些被认可的安全编码器，例如:

1. HTML escape
2. URL encode
3. 某些 SQL 安全包装

引擎会加上如 `XSS_SAFE`、`SQL_INJECTION_SAFE` 等标签，降低风险或不报。

### 第 9 步: 生成 BugInstance

检测器最终会创建 `BugInstance`，里面包含:

1. 漏洞类型
2. 优先级
3. 所在类、方法、源码行
4. 额外上下文

### 第 10 步: 输出结果

SpotBugs/FindSecBugs 最终把这些结果以文本、XML 或其他可处理格式输出。

## 一个最容易懂的例子: SQL 注入

假设你的代码大致是:

```java
String id = request.getParameter("id");
String sql = "select * from user where id = " + id;
statement.executeQuery(sql);
```

FindSecBugs 在脑子里做的事，可以粗略想成:

1. `request.getParameter("id")` 来自用户输入，所以 `id` 被标为 tainted
2. `sql` 由 tainted 数据拼接得到，所以 `sql` 也 tainted
3. `Statement.executeQuery(String)` 的字符串参数是 SQL sink
4. tainted 数据进了 SQL sink
5. 报 `SQL_INJECTION_JDBC`

这就是 taint 分析的最核心思路。

## 再看一个简单例子: 弱哈希

如果代码是:

```java
MessageDigest.getInstance("MD5");
```

它不需要做整条数据流分析，而是更像:

1. 看见调用 `MessageDigest.getInstance`
2. 看到参数常量是 `MD5`
3. 直接报 `WEAK_MESSAGE_DIGEST_MD5`

所以你可以把 FindSecBugs 理解成:

1. 一部分规则是“模式匹配”
2. 一部分规则是“数据流分析”

## 为什么会有误报和漏报

静态分析几乎一定有误报和漏报，这是原理决定的。

### 误报来源

1. 工具看不到运行时某些约束
2. 某个自定义净化函数没被配置成 safe encoder
3. 某个封装 API 行为被工具保守处理

### 漏报来源

1. 依赖类不完整
2. 自定义框架入口没被识别成 source
3. 自定义危险 API 没加入 sink
4. 复杂反射、动态代理、运行时拼装路径超出静态可见范围

所以正确姿势不是“迷信它”，而是:

1. 让它先帮你缩小排查范围
2. 再用人工判断做最终确认

## 它的工作流为什么适合工程化

因为这套流程有很强的批处理特征:

1. 输入是编译产物
2. 规则是机器可重复执行的
3. 结果是结构化的
4. 可以稳定集成进 CI

这就是为什么它很适合:

1. 日常构建扫描
2. 版本回归对比
3. 大批量项目审计

## 非科班用户的类比

可以把它想成“快递安检中的路径追踪”:

1. 来自外部的包裹先贴上“未知来源”标签
2. 包裹在仓库里被搬来搬去
3. 如果它最后进了“危险区域”，就拉响警报
4. 如果中途经过了标准消杀流程，风险可以下降

这就是 taint 分析最直观的现实类比。

## 你该记住的 5 个关键词

1. `字节码`
2. `检测器`
3. `source`
4. `sink`
5. `taint`

把这五个词理顺了，你对 FindSecBugs 的理解就已经超过“只会运行命令”的阶段了。
