# Find Security Bugs 的规则仓库与规则组织

## 这份文档回答什么

这份文档回答的是:

1. FindSecBugs 的规则到底存在哪
2. 规则不是一个文件，而是一整套什么结构
3. 一个“漏洞规则”由哪些部分拼起来
4. 如果你要扩展或排查误报，应该先看哪里

对非科班读者，可以先记住一句:

`FindSecBugs 的“规则库”不是单一规则文件，而是“检测器源码 + 漏洞模式元数据 + 污染源配置 + 危险汇点配置 + 安全编码器配置 + 样例工程”的组合。`

## 要把“规则仓库”拆成 6 个层面来看

### 1. 检测器源码层

位置:

`findsecbugs-plugin/src/main/java/`

作用:

真正写“怎么发现问题”的 Java 代码。

例子:

1. `crypto/WeakMessageDigestDetector.java`
2. `injection/sql/SqlInjectionDetector.java`
3. `xml/XxeDetector.java`
4. `file/PathTraversalDetector.java`

### 2. 插件注册层

位置:

`findsecbugs-plugin/src/main/resources/metadata/findbugs.xml`

作用:

告诉 SpotBugs:

1. 这个插件是谁
2. 要注册哪些检测器
3. 每个检测器会产出哪些漏洞类型
4. 这些漏洞属于什么类别、对应什么 CWE

### 3. 漏洞说明层

位置:

`findsecbugs-plugin/src/main/resources/metadata/messages.xml`

作用:

给每个漏洞模式配上人类可读说明，例如:

1. 这类风险是什么
2. 为什么危险
3. 应该怎么修

可以把它理解成“规则结果的说明书”。

### 4. 污染源与传播层

位置:

`findsecbugs-plugin/src/main/resources/taint-config/`

作用:

告诉引擎:

1. 哪些 API 返回不可信输入
2. 哪些 API 会保留输入污染性
3. 哪些 API 会复制、传播或改变 taint 状态

### 5. 危险汇点层

位置:

`findsecbugs-plugin/src/main/resources/injection-sinks/`

作用:

告诉引擎:

“如果 tainted 数据流到了这些参数位，就要报注入类风险。”

### 6. 安全编码器层

位置:

`findsecbugs-plugin/src/main/resources/safe-encoders/`

作用:

告诉引擎:

“这些函数相当于净化、编码或安全包装，可以降低或消除某些注入风险标签。”

## 元数据文件 `findbugs.xml` 是总目录

如果你不知道从哪开始读规则，优先看:

`findsecbugs-plugin/src/main/resources/metadata/findbugs.xml`

它相当于“插件总清单”。

我本地核验到这个文件里有:

1. `pluginid="com.h3xstream.findsecbugs"`
2. `provider="Find Security Bugs"`
3. `defaultenabled="true"`
4. `website="https://find-sec-bugs.github.io"`
5. `EngineRegistrar` 注册 taint 引擎

并且当前版本源码里注册了:

1. `121` 个检测器
2. `144` 个漏洞模式

这也是官网首页“144 bug patterns”的来源之一。

## 一个规则不是一条 if，而是一整条链

拿 `SQL 注入` 举例，一个规则往往至少包含这些部件:

1. 检测器源码: 例如 `SqlInjectionDetector`
2. sink 文件: 例如 `injection-sinks/sql-jdbc.txt`
3. taint 配置: 识别哪些数据来自请求参数、Header、Cookie 等
4. safe encoder 配置: 识别哪些处理能降低风险
5. BugPattern 元数据: 把结果映射为 `SQL_INJECTION_JDBC` 等类型
6. messages 说明: 给用户解释为什么报、如何修

所以不要把“规则库”理解成传统 IDS/正则那种“一行一条规则”。这里是更偏静态分析工程的组织方式。

## 为什么端点规则也算规则库的一部分

初学者会疑惑:

“入口识别又不是漏洞，为什么也放进来?”

因为静态分析里，入口识别是数据流分析的起点。

如果工具不知道:

1. 请求参数从哪来
2. header 从哪来
3. cookie 从哪来
4. MVC 控制器的参数如何绑定

那它后面就很难准确判断 SQL 注入、XSS、路径遍历等问题。

所以端点与 source 规则其实是整个安全规则库的底座。

## 如何读一个具体规则家族

以 JDBC SQL 注入为例，你可以按这个顺序看:

1. `metadata/findbugs.xml` 里找到 `SqlInjectionDetector`
2. 看 `SqlInjectionDetector.java` 加载了哪些 sink 文件
3. 看 `injection-sinks/sql-jdbc.txt`
4. 看 `taint-config/java-ee.txt` 里请求参数如何被标记为 tainted
5. 看 `safe-encoders/other.txt` 里哪些 SQL 相关函数会被标记为 safe
6. 再去 `messages.xml` 查最终展示说明

这是一条完整“规则阅读路径”。

## 为什么说 `findsecbugs-plugin` 才是规则仓库核心

因为真正决定“能不能报、报什么、为什么报”的内容，全都集中在它里面:

1. Java 检测器
2. XML 元数据
3. txt 规则配置
4. 国际化/消息说明

而 `cli/` 主要解决的是“怎么打包和运行”。

## 样例工程为什么也应视为规则仓库的一部分

很多开源扫描器都有一个问题:

规则是有了，但没有公开、稳定、可重放的样例工程，结果你很难知道一条规则到底期望检测什么。

FindSecBugs 仓库里有多套样例:

1. `findsecbugs-samples-java`
2. `findsecbugs-samples-java11`
3. `findsecbugs-samples-jsp`
4. `findsecbugs-samples-kotlin`

这些样例的重要性在于:

1. 帮助开发者验证规则
2. 帮助使用者理解规则触发条件
3. 帮助你区分“预期命中”和“误报”

所以从学习角度，样例目录也是规则仓库的一部分。

## 规则仓库的“组织哲学”

这个项目的设计思路，不是把所有知识写死在 Java 里，而是分层:

1. 稳定的引擎逻辑写在 Java 类中
2. 易变的 API 识别知识写在资源文件中
3. 面向用户的文案写在元数据/消息中

这样做有三个好处:

1. 新增框架 API 时，不一定每次都改大量核心代码
2. 规则和说明可以分开维护
3. 社区贡献门槛更低

## 非科班读者可以这样理解“规则仓库”

把它想成一本会工作的“安检手册”:

1. Java 检测器是安检员的动作流程
2. `findbugs.xml` 是值班表和检查项目表
3. `taint-config` 是“哪些东西算可疑输入”
4. `injection-sinks` 是“哪些地方一旦接触可疑输入就危险”
5. `safe-encoders` 是“哪些处理可视为消毒”
6. `messages.xml` 是给最终使用者看的解释页

## 学习建议

第一次读时，不要试图一下读完整个仓库。推荐分三轮:

### 第一轮

只读:

1. `README.md`
2. `cli/findsecbugs.sh`
3. `metadata/findbugs.xml`

目标:

知道它是什么、怎么跑、有哪些规则家族。

### 第二轮

只挑 1 个规则家族深挖，例如 SQL 注入:

1. `SqlInjectionDetector.java`
2. `injection-sinks/sql-jdbc.txt`
3. `taint-config/java-ee.txt`

目标:

知道一条实际风险是怎么被报出来的。

### 第三轮

再回来看:

1. `TaintDataflowEngine.java`
2. `TaintMethodConfig.java`
3. 其他注入/XSS/路径遍历规则

目标:

理解整套引擎而不是只会查文件。
